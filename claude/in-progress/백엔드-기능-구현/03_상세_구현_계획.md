## Phase 1 — 커뮤니티 핵심

### 1-A. API 라우트 생성

#### `app/api/posts/route.ts`
```
GET  /api/posts  — 목록 조회 (category, feed, page 쿼리 파라미터)
POST /api/posts  — 게시글 생성 (로그인 필수)
```

**GET 쿼리 파라미터**:
- `category`: review | info | strategy | free | all (기본값 all)
- `feed`: latest | best (기본값 latest)
- `page`: 숫자 (기본값 1, 페이지당 20개)
- `userId`: 선택 — 내 좋아요 여부 포함 시

**응답 포함 필드**:
```typescript
{
  id, category, title, content(앞 100자), 
  like_count, comment_count, view_count, created_at,
  user: { id, nickname, profile_image },
  isLiked: boolean  // 로그인한 경우만
}
```

---

#### `app/api/posts/[id]/route.ts`
```
GET    /api/posts/[id]  — 상세 조회 + 조회수 +1
PATCH  /api/posts/[id]  — 수정 (본인만)
DELETE /api/posts/[id]  — 삭제 (본인만)
```

---

#### `app/api/posts/[id]/like/route.ts`
```
POST   /api/posts/[id]/like  — 좋아요 토글 (로그인 필수)
```
- `post_likes` 행이 있으면 DELETE, 없으면 INSERT
- `posts.like_count` 동시 업데이트

---

#### `app/api/posts/[id]/comments/route.ts`
```
GET  /api/posts/[id]/comments  — 댓글 목록 (대댓글 포함 트리 구조)
POST /api/posts/[id]/comments  — 댓글 작성 (로그인 필수)
```

**댓글 트리 응답**:
```typescript
[
  { id, content, user, created_at, replies: Comment[] }
]
```

---

#### `app/api/comments/[id]/route.ts`
```
DELETE /api/comments/[id]  — 댓글 삭제 (본인만)
```
- 실제 삭제 (hard delete) or 내용 마스킹 처리
- 댓글 삭제 시 `posts.comment_count` 갱신

## Phase 2 — 소셜 기능

### 2-A. 팔로우/언팔로우

**팔로우가 가능한 이유**: `follows` 테이블이 이미 설계되어 있음.
- `follower_id`(나) → `following_id`(상대) INSERT = 팔로우
- DELETE = 언팔로우
- `profiles.follower_count` / `following_count` 캐시 갱신

#### `app/api/users/[userId]/follow/route.ts`
```
POST   — 팔로우 (본인 팔로우 불가 체크)
DELETE — 언팔로우
GET    — 팔로우 여부 확인
```

#### `/u/[nickname]/page.tsx` 수정
- 팔로우/팔로잉 수 표시 (현재 없음)
- 팔로우 버튼 (본인 프로필이면 숨김)
- 버튼 클릭 → API 호출 + 낙관적 업데이트

#### 커뮤니티 팔로우 피드
- `community/page.tsx` "팔로우" 탭 → `/api/posts?feed=following` 연동
- 팔로우한 유저 목록 조회 후 해당 유저들의 게시글만 필터

---

### 2-B. 프로필 수정

#### `app/api/profile/route.ts`
```
PATCH /api/profile  — 닉네임, 한마디(bio) 저장
```
- 닉네임 변경 시 중복 확인
- `profiles` 테이블 UPDATE

#### 이미지 업로드 (`avatars` 버킷)
- 클라이언트에서 Supabase Storage 직접 업로드 (인증된 유저만 가능 — RLS로 보호)
- 업로드 경로: `avatars/{userId}/avatar.jpg`
- 업로드 후 URL을 `profiles.profile_image`에 저장

```typescript
// lib/upload.ts
export async function uploadAvatar(userId: string, file: File): Promise<string> {
  const supabase = createBrowserClient(...)
  const { data } = await supabase.storage
    .from('avatars')
    .upload(`${userId}/avatar.jpg`, file, { upsert: true })
  return supabase.storage.from('avatars').getPublicUrl(data.path).data.publicUrl
}
```

#### `profile/edit/page.tsx` 연동
- 이미지 업로드 → URL 저장
- 닉네임/bio 저장
- 성공 토스트

### 3-C. 플레이 로그 CRUD

#### `app/api/play-logs/route.ts`
```
POST /api/play-logs  — 로그 생성
```

#### `app/api/play-logs/[id]/route.ts`
```
GET    — 로그 상세 (본인 or 공개인 경우)
PATCH  — 수정 (본인만)
DELETE — 삭제 (본인만)
```

#### DB 마이그레이션 필요
```sql
-- play_logs에 visibility 컬럼 추가
ALTER TABLE play_logs 
ADD COLUMN visibility TEXT NOT NULL DEFAULT 'public'
  CHECK (visibility IN ('public', 'private'));
```

```sql
-- RLS 정책 업데이트
DROP POLICY "play_logs_select" ON play_logs;
CREATE POLICY "play_logs_select" ON play_logs FOR SELECT
  USING (user_id = auth.uid() OR visibility = 'public');
```

#### LogForm 연동
- `my/logs/write/page.tsx`: `LogForm.onSubmit` → `POST /api/play-logs`
- `my/logs/[id]/edit/page.tsx`: `LogForm.onSubmit` → `PATCH /api/play-logs/[id]`
- `my/logs/[id]/page.tsx`: 삭제 버튼 → `DELETE /api/play-logs/[id]`
- LogForm에 "공개 범위" 토글 추가 (공개/나만보기)
