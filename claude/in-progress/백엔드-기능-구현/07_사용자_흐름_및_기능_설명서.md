> 작성일: 2026-04-13  
> 대상: 기획자, 디자이너, 신규 개발자  
> 목적: 코드 없이 "누가, 무엇을, 어떻게 쓸 수 있는지" 전체 그림 파악

## 1. 사용자 유형과 권한

BoardLog의 사용자는 세 가지 역할로 구분됩니다.

### 비회원
서비스에 로그인하지 않은 상태입니다.

- 게임 목록·상세 정보 조회 가능
- 커뮤니티 게시글·댓글 읽기 가능
- 다른 유저의 공개 프로필·공개 플레이로그 열람 가능
- 좋아요 여부는 항상 false로 표시 (좋아요 수는 보임)
- 쓰기 행위(글 작성, 좋아요, 댓글, 팔로우, 컬렉션, 평점, 로그 작성) 전부 불가
- 쓰기 시도 시 로그인 유도 시트(LoginPromptSheet)가 노출됨

### 회원 (로그인 상태)
소셜 로그인 또는 이메일로 가입한 상태입니다.

- 비회원 권한 전체 포함
- 게시글 작성·수정·삭제 (본인 글만)
- 댓글·대댓글 작성·삭제 (본인 댓글만)
- 좋아요 토글
- 다른 유저 팔로우·언팔로우
- 게임 컬렉션 상태 저장 (소유 중 / 해보고 싶어요 / 즐겨찾기 등)
- 게임 평점 저장·삭제
- 플레이 로그 작성·수정·삭제 (본인 로그만)
- 비공개 로그는 본인만 조회 가능
- 프로필(닉네임, 소개, 프로필 이미지) 수정

### 본인
로그인된 본인의 리소스에 접근하는 경우입니다.

- 본인 게시글·댓글·로그만 수정·삭제 가능
- 비공개(private) 플레이 로그 열람 가능
- 타인의 비공개 로그 접근 시 403 반환

## 2. 접근 제어 구조

권한 검증은 두 단계로 이루어집니다.

### 1단계 — API 서버 검증
모든 쓰기 요청은 서버에서 인증 토큰을 확인합니다.
- 토큰 없음 → 401 Unauthorized
- 타인 리소스 수정 시도 → 403 Forbidden
- 존재하지 않는 리소스 → 404 Not Found
- 중복 요청(팔로우 중인데 팔로우) → 409 Conflict

### 2단계 — Supabase RLS(Row Level Security)
DB 레벨에서 마지막 방어선 역할을 합니다. API를 우회해 직접 DB에 접근해도 정책에 따라 데이터가 보호됩니다.

## 3. 사용자 흐름

### 3-1. 비회원 탐색 흐름

```
앱 진입
  └─ 게임 목록 탐색 → 게임 상세 보기 (평점·설명·인원·시간)
  └─ 커뮤니티 탐색 → 게시글 읽기 → 댓글 읽기
  └─ 유저 프로필 방문 → 공개 플레이로그 열람
  └─ 쓰기 시도 → 로그인 유도 시트 노출 → 로그인/회원가입 페이지로 이동
```

### 3-2. 회원가입 / 로그인 흐름

```
로그인 페이지
  └─ 이메일+비밀번호 입력 → Supabase Auth 처리
  └─ 성공 → 이전 페이지로 복귀 (callbackUrl 파라미터 활용)
  └─ 실패 → 에러 메시지 노출

회원가입 페이지
  └─ 이메일+비밀번호+닉네임 입력 → Supabase Auth 처리
  └─ 가입 완료 → 자동 로그인 → 홈으로 이동

로그인 완료 시
  └─ authStore에 user·profile 저장
  └─ 백그라운드에서 내 컬렉션·평점 fetch → collectionStore 초기화
  └─ 이후 게임 상세 진입 시 기존 컬렉션 상태·평점 즉시 표시
```

### 3-3. 커뮤니티 흐름

#### 게시글 목록
```
커뮤니티 탭 진입
  └─ 피드 선택: 최신 / 인기 / 팔로잉
       ├─ 최신: 전체 게시글 최신순
       ├─ 인기: 좋아요 수 기준 정렬
       └─ 팔로잉: 내가 팔로우한 유저 게시글만 (비로그인 시 최신으로 폴백)
  └─ 카테고리 필터: 전체 / 후기 / 정보 / 공략 / 자유
  └─ 게시글 카드 클릭 → 상세 페이지
```

#### 게시글 상세
```
게시글 상세 진입
  └─ 게시글 내용 + 작성자 + 조회수 표시
       └─ 조회수는 진입 시 자동으로 +1 (비동기, 페이지 응답에는 영향 없음)
  └─ 좋아요 버튼
       ├─ 비로그인 → 로그인 유도
       └─ 로그인 → 토글 (좋아요 추가/취소, 카운트 즉시 반영)
  └─ 댓글 목록
       ├─ 2단 트리 구조: 댓글 > 대댓글 (1단 깊이 제한)
       └─ 댓글 작성자 옆 답글 버튼 → 대댓글 입력창 활성화
  └─ 본인 글이면 수정·삭제 버튼 노출
```

#### 게시글 작성
```
글쓰기 버튼 클릭
  └─ 비로그인 → 로그인 유도
  └─ 로그인 → 작성 페이지
       └─ 카테고리 선택 + 제목(1~100자) + 내용(1~5000자) 입력
       └─ 제출 → 저장 완료 → 새 게시글 상세 페이지로 이동
```

### 3-4. 소셜 (팔로우) 흐름

```
다른 유저 프로필 방문
  └─ 팔로우 버튼 노출 (본인 프로필이면 숨김)
       ├─ 비로그인 → 로그인 유도
       ├─ 팔로우 중 → 버튼에 "팔로잉" 표시, 클릭 시 언팔로우
       └─ 팔로우 안 함 → 버튼에 "팔로우" 표시, 클릭 시 팔로우
  └─ 팔로우/언팔로우 즉시 팔로워·팔로잉 수 갱신

자신의 팔로우 시도 → 400 에러 (UI에서 버튼 자체를 숨겨 방지)
이미 팔로우 중인데 팔로우 → 409 에러 처리
```

### 3-5. 게임 컬렉션·평점 흐름

```
게임 상세 페이지 진입
  └─ 로그인 상태이면: 내 컬렉션 상태·평점이 즉시 표시
       (로그인 시점에 이미 서버에서 불러와 저장해둔 데이터)

컬렉션 버튼 클릭
  └─ 비로그인 → 로그인 유도
  └─ 로그인 → CollectionSheet 열림
       └─ 상태 선택: 소유 중 / 찜 / 즐겨찾기 / 해봤어요 / 플레이 중 등
       └─ 선택 즉시 화면에 반영 (낙관적 업데이트)
       └─ 백그라운드에서 서버 저장 → 실패 시 이전 상태로 복원

평점 버튼 클릭
  └─ 비로그인 → 로그인 유도
  └─ 로그인 → 별점 모달 열림
       └─ 별점 선택 (1~10점, 0.5 단위)
       └─ 저장 → 즉시 반영 + 게임 평균 평점·평가 수 서버 재계산
       └─ 기존 평점 있으면 삭제 버튼도 노출
```

### 3-6. 플레이 로그 흐름

#### 로그 작성
```
기록 작성 페이지 진입 (비로그인 시 로그인 유도)
  └─ 게임 선택 (검색 시트)
  └─ 플레이 날짜 선택
  └─ 함께한 사람 입력 (선택, 칩 UI로 여러 명 추가)
  └─ 장소 입력 (선택)
  └─ 내 평점 선택 (선택, 별점 UI)
  └─ 메모 입력 (선택, 자유 텍스트)
  └─ 공개 범위: 전체 공개 / 나만 보기
  └─ 저장 → 로그 목록으로 이동
```

#### 로그 상세 / 수정 / 삭제
```
로그 상세 진입
  └─ 비공개 로그: 본인만 접근 가능 (타인 접근 시 403)
  └─ 본인이면: 수정·삭제 버튼 노출
       ├─ 수정: 기존 데이터로 폼 초기화 → 수정 완료 → 상세로 복귀
       └─ 삭제: 즉시 삭제 → 로그 목록으로 이동
```

### 3-7. 프로필 수정 흐름

```
프로필 편집 페이지
  └─ 현재 닉네임·소개·프로필 이미지로 폼 초기화
  └─ 이미지 변경
       └─ 파일 선택 → 브라우저에서 직접 Supabase Storage에 업로드
       └─ 업로드 완료 → 이미지 URL을 프로필 수정 요청에 포함
  └─ 닉네임 수정 (2~20자, 중복 불가 검증)
  └─ 저장 → authStore 세션 갱신 → 변경사항 즉시 앱 전체에 반영
```

## 4. 기능별 구현 방식

### 인증 (Authentication)
Supabase Auth를 사용합니다. 로그인 상태는 앱 전역에서 `authStore`(Zustand)로 관리됩니다.

- 앱 첫 로드 시 현재 세션 확인 → 로그인 상태 복원
- 로그인 성공 시: 프로필 정보 + 내 컬렉션·평점을 백그라운드 fetch
- 세션 변경(로그인·로그아웃) 감지 → 자동으로 상태 업데이트
- 로그아웃 시: 컬렉션·평점 스토어도 함께 초기화

### 상태 관리 (State Management)
두 개의 전역 스토어를 사용합니다.

**authStore**
- 현재 로그인 사용자 정보(user, profile) 보관
- isLoggedIn, isLoading 플래그로 UI 분기

**collectionStore**
- 게임 ID별 컬렉션 상태·평점을 메모리에 캐싱
- 게임 상세 페이지에서 API 없이 즉시 내 상태 표시 가능
- 로그인 시 서버에서 가져온 전체 데이터로 초기화

### 낙관적 업데이트 (Optimistic Update)
컬렉션·평점 변경 시 서버 응답을 기다리지 않고 UI를 먼저 바꿉니다.

1. 버튼 클릭 → 화면에 즉시 새 상태 반영
2. 백그라운드에서 서버에 저장 요청
3. 성공 → 그대로 유지
4. 실패 → 이전 상태로 롤백 + 에러 처리

사용자 입장에서는 지연 없이 즉각적인 반응을 경험합니다.

### 데이터 캐싱 (React Query)
서버 데이터는 React Query로 캐싱합니다.

- 게시글 목록, 상세, 댓글은 캐시키로 관리
- mutation(작성·삭제·좋아요) 성공 시 관련 캐시 무효화 → 자동 재조회
- staleTime 설정으로 불필요한 재조회 방지

### 댓글 트리 구조
DB에는 댓글이 평면(flat)으로 저장됩니다. `parent_id` 컬럼으로 대댓글 관계를 표현합니다.

- 서버에서 해당 게시글의 모든 댓글을 한 번에 조회
- 클라이언트에서 ID 맵을 만들어 트리로 변환 (O(n) 처리)
- UI에서는 댓글 > 대댓글 2단 구조로 렌더링
- 대댓글에는 대댓글을 달 수 없음 (1단 깊이 제한)

### 이미지 업로드
프로필 이미지는 서버를 경유하지 않고 브라우저에서 직접 Supabase Storage에 업로드합니다.

- 업로드 경로: `avatars/{userId}/{timestamp}.jpg`
- 업로드 완료 후 이미지 URL을 프로필 수정 API에 전달
- 기존 이미지는 upsert 방식으로 덮어씌워짐

### 평균 평점 계산
게임 평점 저장·삭제 시 해당 게임의 전체 평점을 재계산합니다.

- 해당 게임의 모든 평점을 조회
- 평균값·개수를 계산해 games 테이블의 avg_rating, rating_count 업데이트
- 게임 상세 페이지의 "커뮤니티 평점"에 반영

### 팔로우 수 갱신
팔로우·언팔로우 시 follows 테이블의 실제 행 수를 세어 profiles 테이블을 업데이트합니다.

- 캐시된 카운트 대신 COUNT 쿼리로 정확한 값 유지
- 팔로워 수와 팔로잉 수 모두 갱신

## 5. 공개/비공개 처리

### 플레이 로그
`visibility` 필드로 공개 여부를 결정합니다.

| 상태 | 내 로그 | 타인 로그 |
|------|---------|----------|
| public | 조회 가능 | 조회 가능 |
| private | 조회 가능 | 403 반환 |

DB의 RLS 정책이 이를 강제합니다. API를 우회해도 DB에서 차단됩니다.

### 커뮤니티 게시글
현재 모든 게시글은 공개입니다. 비공개 게시글 기능은 구현되지 않았습니다.

## 6. 에러 처리 흐름

| 상황 | 응답 | 사용자 경험 |
|------|------|------------|
| 로그인 필요한 작업 | 401 | 로그인 유도 시트 노출 |
| 타인 리소스 접근 | 403 | "권한이 없습니다" 메시지 |
| 없는 리소스 | 404 | "찾을 수 없습니다" 메시지 |
| 중복 요청 | 409 | 무시 또는 안내 메시지 |
| 유효성 검사 실패 | 400 | 입력 필드 에러 메시지 |
| 서버 오류 | 500 | "오류가 발생했습니다" 메시지 |
| 낙관적 업데이트 실패 | - | 이전 상태로 자동 복원 |

## 7. 현재 구현되지 않은 기능

서비스 출시 후 추가 개발이 필요한 항목들입니다.

| 기능 | 위치 | 설명 |
|------|------|------|
| 게시글 이미지 첨부 | 커뮤니티 글쓰기 | Storage 연동 미구현, 버튼 UI만 존재 |
| 게시글 게임 태그 | 커뮤니티 글쓰기 | 게임을 태그하는 기능 미구현 |
| 공유 기능 | 게시글 상세 | Web Share API 미연결 |
| 게임 검색 실연동 | 로그 작성 | Mock 검색, 실제 DB 검색 필요 |
| 알림 기능 | 전체 | 팔로우·댓글·좋아요 알림 미구현 |
| 인기 피드 고도화 | 커뮤니티 | 현재 단순 좋아요 수 정렬, 시간 가중치 미적용 |
| 평점 race condition | 평점 저장 | 동시 요청 시 avg_rating 오차 가능, DB 트리거 도입 권장 |

## 8. 데이터 흐름 요약

```
사용자 액션
    │
    ▼
클라이언트 (React + Zustand + React Query)
    │  ├─ 즉시 UI 반영 (낙관적 업데이트)
    │  └─ 캐시 관리 (staleTime, invalidation)
    │
    ▼
Next.js Route Handler (서버)
    │  ├─ requireUser() → 인증 검증
    │  ├─ 소유권 확인 → 권한 검증
    │  └─ 비즈니스 로직 처리
    │
    ▼
Supabase
    ├─ RLS 정책 (최후 방어선)
    ├─ PostgreSQL DB
    └─ Storage (이미지)
```
