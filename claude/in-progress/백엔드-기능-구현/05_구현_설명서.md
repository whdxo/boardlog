## 사전 작업 — DB 상태 확인

### 왜 필요한가
UI와 타입은 완성돼 있지만, Supabase에 실제 테이블이 존재하는지 확인된 적 없다.
`supabase/migrations/` 파일은 설계서일 뿐이고, 대시보드에 적용했는지가 별개 문제다.
API를 아무리 잘 만들어도 테이블이 없으면 전부 에러가 난다.

| 확인 항목 | 난이도 |
|-----------|--------|
| posts, comments, follows, collections, ratings, play_logs 테이블 존재 여부 | ⭐ |
| avatars, post-images Storage 버킷 존재 여부 | ⭐ |

**없으면**: Supabase SQL Editor에서 migration 파일을 직접 실행해야 한다.

## Phase 1 — 커뮤니티 핵심

지금 커뮤니티 화면은 전부 하드코딩된 목업 데이터로 보여주고 있다.
이 Phase가 끝나면 실제 유저가 글을 쓰고, 읽고, 댓글 달고, 좋아요를 누를 수 있게 된다.

---

### 1-1. 게시글 목록 API

**무엇을 하나**: 커뮤니티 메인 화면에서 게시글 목록을 DB에서 불러온다.  
카테고리(리뷰/정보/전략/자유), 정렬(최신/베스트), 페이지 번호를 조건으로 받아서 필터링한 결과를 반환한다.  
로그인한 유저라면 각 게시글에 내가 좋아요를 눌렀는지 여부도 함께 내려준다.

**왜 까다롭지 않은가**: 조건이 많아 보이지만 쿼리 파라미터를 Supabase 쿼리에 연결하는 표준 패턴이다.

| 작업 | 난이도 |
|------|--------|
| 카테고리/정렬/페이지 필터 쿼리 | ⭐⭐ |
| 내 좋아요 여부 포함 (로그인 시) | ⭐⭐ |

---

### 1-2. 게시글 생성 API

**무엇을 하나**: 커뮤니티 글쓰기 폼에서 "발행" 버튼을 누르면 DB에 저장한다.  
로그인하지 않은 유저는 거부하고, 저장 성공 시 새 게시글 ID를 반환해서 상세 페이지로 이동시킨다.

| 작업 | 난이도 |
|------|--------|
| 인증 확인 + posts 테이블 INSERT | ⭐ |
| community/new 페이지에 연결 | ⭐ |

---

### 1-3. 게시글 상세 API

**무엇을 하나**: 글 하나를 불러오고, 동시에 조회수를 1 올린다.  
수정/삭제는 작성자 본인만 가능하도록 막는다.

**주의점**: 조회수 갱신을 매번 정확히 할 필요는 없고, 같은 사람이 새로고침할 때마다 오르는 것도 일단 허용 범위다.

| 작업 | 난이도 |
|------|--------|
| 게시글 상세 조회 + view_count +1 | ⭐ |
| 수정/삭제 본인 확인 | ⭐ |
| community/[id] 페이지에 연결 | ⭐⭐ |

---

### 1-4. 좋아요 토글

**무엇을 하나**: 좋아요 버튼을 누르면 눌린 상태 ↔ 안 눌린 상태를 전환한다.  
이미 좋아요한 상태면 취소, 아니면 추가. posts 테이블의 like_count 숫자도 함께 갱신한다.

**왜 중요한가**: 화면에서 즉시 반응하는 것처럼 느껴져야 하기 때문에 낙관적 업데이트(API 응답 전에 UI를 먼저 바꾸는 것)가 필요하다.

| 작업 | 난이도 |
|------|--------|
| 좋아요 INSERT/DELETE 토글 로직 | ⭐ |
| like_count 갱신 | ⭐ |
| UI 낙관적 업데이트 | ⭐⭐ |

---

### 1-5. 댓글 시스템

**무엇을 하나**: 댓글 목록을 불러오고, 새 댓글을 저장하고, 삭제한다.  
대댓글(1depth)을 지원하므로 목록을 단순 배열이 아닌 트리 형태로 조합해서 반환해야 한다.

**왜 까다로운가**: DB에서는 댓글이 flat하게 저장되지만(parent_id 컬럼으로 연결), 화면에 보여줄 때는 부모 댓글 아래에 자식을 묶어서 트리로 조합해야 한다. 이 변환 로직이 핵심이다.

| 작업 | 난이도 |
|------|--------|
| 댓글 목록 트리 구조 조합 | ⭐⭐⭐ |
| 댓글 작성 (대댓글 포함) | ⭐⭐ |
| 댓글 삭제 + comment_count 갱신 | ⭐⭐ |
| 댓글 UI 연결 | ⭐⭐ |

## Phase 2 — 소셜 기능

커뮤니티가 완성된 뒤, 유저 간 연결을 만드는 단계다.

---

### 2-1. 팔로우 / 언팔로우

**무엇을 하나**: 다른 유저의 프로필 페이지에서 팔로우 버튼을 누르면 관계가 생긴다.  
팔로우하면 커뮤니티의 "팔로우" 탭에 그 유저의 게시글이 보인다.  
프로필에는 팔로워/팔로잉 수가 표시된다.

**왜 어렵지 않은가**: `follows` 테이블에 행을 추가하거나 지우는 단순 구조다. 본인을 팔로우하는 것만 막으면 된다.

| 작업 | 난이도 |
|------|--------|
| 팔로우/언팔로우 API | ⭐ |
| 프로필 페이지 팔로우 버튼 연결 | ⭐⭐ |
| 팔로워/팔로잉 수 표시 | ⭐ |
| 커뮤니티 팔로우 피드 연동 | ⭐⭐ |

---

### 2-2. 프로필 수정

**무엇을 하나**: 닉네임, 한마디(bio), 프로필 사진을 바꿀 수 있게 한다.  
이미지는 Supabase Storage에 올리고 URL을 DB에 저장한다.  
닉네임은 중복 여부를 먼저 확인해야 한다.

| 작업 | 난이도 |
|------|--------|
| 닉네임/bio 저장 API | ⭐ |
| 닉네임 중복 체크 | ⭐⭐ |
| 프로필 이미지 업로드 (Storage) | ⭐⭐ |
| profile/edit 페이지 연결 | ⭐⭐ |

---

## Phase 3 — 개인 기록

이미 UI가 완성돼 있다. API를 만들고 버튼에 연결하는 것이 주 작업이다.

---

### 3-1. 플레이 로그 공개 범위 마이그레이션

**무엇을 하나**: 플레이 로그를 공개/비공개로 설정할 수 있게 DB 구조를 먼저 바꾼다.  
`play_logs` 테이블에 `visibility` 컬럼을 추가하고, RLS(행 수준 보안) 정책도 함께 수정한다.

**왜 먼저 해야 하나**: 나중에 추가하면 기존 데이터 처리가 복잡해진다. 로그 저장 API를 만들기 전에 스키마를 확정해야 한다.

| 작업 | 난이도 |
|------|--------|
| play_logs 테이블 컬럼 추가 마이그레이션 | ⭐⭐ |
| RLS 정책 업데이트 | ⭐⭐ |

---

### 3-2. 컬렉션 저장

**무엇을 하나**: 게임 상세 페이지의 `CollectionSheet`에서 "소장 중", "위시리스트" 등 상태를 선택하면 DB에 저장된다.  
같은 게임을 다시 선택하면 상태가 업데이트(upsert)된다.

| 작업 | 난이도 |
|------|--------|
| 컬렉션 추가/수정 API (upsert) | ⭐ |
| 컬렉션 제거 API | ⭐ |
| CollectionSheet 저장 연결 | ⭐⭐ |

---

### 3-3. 평점 저장

**무엇을 하나**: 게임 상세 페이지의 별점 UI에서 점수를 매기면 DB에 저장된다.  
평점을 저장할 때마다 해당 게임의 평균 점수(`games.avg_rating`)도 다시 계산해서 업데이트한다.

| 작업 | 난이도 |
|------|--------|
| 평점 저장/수정 API (upsert) | ⭐ |
| games.avg_rating 재계산 갱신 | ⭐⭐ |
| RatingModal 저장 연결 | ⭐ |

---

### 3-4. 플레이 로그 CRUD

**무엇을 하나**: 로그 작성 폼에서 게임 기록을 저장하고, 수정하고, 삭제한다.  
visibility(공개/비공개) 선택 UI도 폼에 추가해야 한다.  
점수 기록은 `play_log_scores` 별도 테이블에도 같이 저장해야 한다.

| 작업 | 난이도 |
|------|--------|
| 플레이 로그 생성 API | ⭐⭐ |
| 플레이 로그 수정/삭제 API | ⭐ |
| LogForm에 공개 범위 UI 추가 | ⭐⭐ |
| 로그 작성/수정/삭제 페이지 연결 | ⭐⭐ |

---

## Phase 4 — 부가 기능

Phase 3까지 완료된 뒤 진행. 없어도 서비스는 동작한다.

---

### 4-1. 이미지 업로드 (공통)

**무엇을 하나**: 커뮤니티 글 작성 시 사진 첨부, 프로필 이미지 업로드(Phase 2에서 선행 구현)를 공통 훅으로 통합한다.  
파일 압축 → Storage 업로드 → URL 반환의 흐름을 재사용 가능한 형태로 만든다.

| 작업 | 난이도 |
|------|--------|
| useImageUpload 공통 훅 | ⭐⭐ |
| post-images 버킷 연동 | ⭐⭐ |

---

### 4-2. 셀렉션 관리

**무엇을 하나**: 내가 큐레이션한 게임 목록(셀렉션)을 생성/수정/삭제하고, 게임을 추가하거나 순서를 바꿀 수 있게 한다.

| 작업 | 난이도 |
|------|--------|
| 셀렉션 CRUD API | ⭐⭐ |
| 게임 추가/제거/순서 변경 | ⭐⭐⭐ |
| my/selections 페이지 연결 | ⭐⭐ |
